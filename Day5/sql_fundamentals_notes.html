SQL Fundamentals
for Absolute Beginners

SQL (Structured Query Language) is the universal language for talking to databases. Every data engineer, analyst, and backend developer uses it daily. By the end of this session you will write real queries with confidence.

What exactly is SQL?

Before we write code, let's understand the big picture.

Imagine a database as a collection of spreadsheet-like tables, where each table stores one type of information. SQL is the language you use to create those tables, put data in them, and ask questions about the data.

🏗️

DDL — Data Definition Language

Builds the blueprint. Creates, alters, or drops tables and their structure.

✏️

DML — Data Manipulation Language

Fills the blueprint with data. Inserts, updates, and deletes rows.

🔍

DQL — Data Query Language

Asks questions. Retrieves data using SELECT and its many clauses.

💡
Analogy for freshers

Think of a database like a school. DDL = building the classrooms. DML = filling them with students. DQL = asking "which students scored above 80?"

The SQL statement skeleton

Almost every SQL query you will ever write follows this order. You don't have to use all clauses — start with what you need.

SQL is not case-sensitive — but the convention is to write keywords in UPPERCASE and table/column names in lowercase. This makes queries much easier to read.

DDL 01 / 06

DDL — Data Definition Language

Commands that define and manage the structure of your database — tables, columns, data types, and constraints.

CommandWhat it doesSimple analogy
CREATE TABLECreates a new table with defined columnsDrawing a blank form / spreadsheet
ALTER TABLEAdds, modifies, or drops a columnAdding or renaming a column in your spreadsheet
DROP TABLEPermanently deletes the table and all its dataShredding the entire spreadsheet
TRUNCATEDeletes all rows but keeps the table structureErasing all entries but keeping the form

CREATE TABLE — building your first table

Let's create a students table. A table definition lists every column with its name and data type.

create_students.sql DDL 
-- Create a table to store student records
CREATE TABLE students (
    student_id   INT          PRIMARY KEY,   -- unique identifier
    first_name   VARCHAR(50)  NOT NULL,      -- cannot be empty
    last_name    VARCHAR(50)  NOT NULL,
    email        VARCHAR(100) UNIQUE,        -- no two students share an email
    age          INT          CHECK (age >= 16),
    grade        CHAR(1),                      -- 'A', 'B', 'C' etc.
    enrolled_at  DATE         DEFAULT CURRENT_DATE
);

Common data types

Data TypeStoresExample
INTWhole numbers42, -7, 1000
VARCHAR(n)Text up to n characters'Alice', 'New Delhi'
CHAR(n)Fixed-length text'A', 'IN'
DECIMAL(p,s)Precise decimal numbers9.99, 1250.50
DATECalendar date'2024-06-15'
BOOLEANTrue or FalseTRUE, FALSE
TEXTUnlimited length textLong descriptions, bios

Constraints — the rules your data must follow

🔑 PRIMARY KEY

Uniquely identifies every row. Never NULL. Every table should have one.

🔗 FOREIGN KEY

Links a column in this table to the PRIMARY KEY of another table. Prevents orphan records.

🚫 NOT NULL

Column must always have a value. Can never be left empty on INSERT.

✅ UNIQUE

All values in this column must be different. Good for emails, usernames.

ALTER TABLE — changing structure after creation

alter_table.sql DDL 
-- Add a new column
ALTER TABLE students
    ADD COLUMN phone VARCHAR(15);

-- Rename a column
ALTER TABLE students
    RENAME COLUMN grade TO final_grade;

-- Drop (remove) a column
ALTER TABLE students
    DROP COLUMN phone;
⚠️
DROP TABLE is permanent!

Running DROP TABLE students deletes the table AND all its data — forever. Always double-check before dropping. In production, always take a backup first.

DML 02 / 06

DML — Data Manipulation Language

Commands that add, change, and remove the actual data inside your tables.

CommandWhat it does
INSERT INTOAdds new rows to a table
UPDATEChanges existing values in one or more rows
DELETE FROMRemoves specific rows from a table

INSERT INTO — adding rows

insert_students.sql DML 
-- Insert a single student
INSERT INTO students (student_id, first_name, last_name, email, age, grade)
VALUES (1, 'Priya', 'Sharma', 'priya@example.com', 20, 'A');

-- Insert multiple students at once
INSERT INTO students (student_id, first_name, last_name, email, age, grade)
VALUES
    (2, 'Arjun',  'Mehta',  'arjun@example.com',  22, 'B'),
    (3, 'Sneha',  'Patel',  'sneha@example.com',  19, 'A'),
    (4, 'Rahul',  'Gupta',  'rahul@example.com',  21, 'C'),
    (5, 'Anjali', 'Singh',  'anjali@example.com', 23, 'B');

UPDATE — changing existing data

update_students.sql DML 
-- Update one student's grade
UPDATE students
SET    grade = 'A'
WHERE  student_id = 4;

-- Update multiple columns at once
UPDATE students
SET    grade = 'B',
       age   = age + 1   -- increment age by 1
WHERE  first_name = 'Arjun';
🛑
ALWAYS use WHERE with UPDATE and DELETE

Without a WHERE clause, UPDATE students SET grade = 'F' will change every single student to grade F. Always be specific about which rows to change.

DELETE FROM — removing rows

delete_students.sql DML 
-- Delete one specific student
DELETE FROM students
WHERE student_id = 5;

-- Delete all students with grade 'C'
DELETE FROM students
WHERE grade = 'C';

-- Delete ALL rows (but table structure remains!)
DELETE FROM students;

DELETE vs TRUNCATE

DELETE removes rows one by one (can use WHERE, can be rolled back). TRUNCATE wipes all rows instantly — faster but cannot be undone as easily.

Transactions protect you

Wrap dangerous operations in BEGIN … COMMIT. If something goes wrong, run ROLLBACK to undo everything. This is your safety net.

DQL 03 / 06

DQL — Data Query Language

The art of asking questions. SELECT is the most powerful and most-used SQL command — you will write it hundreds of times.

The SELECT statement

select_examples.sql DQL 
-- Get ALL columns from students
SELECT *
FROM   students;

-- Get SPECIFIC columns only
SELECT first_name, last_name, grade
FROM   students;

-- Use an alias to rename a column in the output
SELECT first_name   AS name,
       grade        AS score
FROM   students;

-- Get only unique values (no duplicates)
SELECT DISTINCT grade
FROM   students;

-- Limit results to the first 3 rows
SELECT *
FROM   students
LIMIT  3;

Aggregate functions — summarising data

These functions collapse many rows into one summary value. Essential for reporting.

aggregates.sql DQL 
SELECT
    COUNT(*)         AS total_students,   -- how many rows?
    AVG(age)          AS average_age,
    MAX(age)          AS oldest_student,
    MIN(age)          AS youngest_student
FROM students;

-- COUNT per grade using GROUP BY
SELECT   grade, COUNT(*) AS num_students
FROM     students
GROUP BY grade
HAVING   COUNT(*) > 1;   -- only grades with more than 1 student
✅
WHERE vs HAVING — the key difference

WHERE filters individual rows before grouping. HAVING filters groups after aggregation. Rule: if you're filtering on an aggregate function (COUNT, SUM, AVG…), use HAVING.

FunctionWhat it returnsExample
COUNT(*)Total number of rowsCOUNT(*) → 4
SUM(col)Total of all numeric valuesSUM(age) → 84
AVG(col)Average of numeric valuesAVG(age) → 21.0
MAX(col)Highest valueMAX(age) → 23
MIN(col)Lowest valueMIN(age) → 19
Filtering 04 / 06

Filtering — WHERE Clause

WHERE narrows down results. Without it you get every row; with it you get exactly what you need.

Comparison operators

=Equal to
!=Not equal to
<Less than
>Greater than
<=Less or equal
>=Greater or equal
LIKEPattern match
INList of values
filtering.sql WHERE 
-- Simple equality
SELECT * FROM students
WHERE grade = 'A';

-- Range filter
SELECT first_name, age FROM students
WHERE age BETWEEN 18 AND 21;

-- Multiple values with IN
SELECT first_name, grade FROM students
WHERE grade IN ('A', 'B');

-- Pattern matching with LIKE
--   %  = any number of characters
--   _  = exactly one character
SELECT * FROM students
WHERE first_name LIKE 'A%';    -- names starting with A

SELECT * FROM students
WHERE email LIKE '%@example.com';  -- emails from example.com

-- Check for NULL values
SELECT * FROM students
WHERE grade IS NULL;   -- never use = NULL, always IS NULL

Combining conditions — AND / OR / NOT

logical_operators.sql WHERE 
-- Both conditions must be true (AND)
SELECT * FROM students
WHERE age >= 20
  AND grade = 'A';

-- Either condition can be true (OR)
SELECT * FROM students
WHERE grade = 'A'
   OR grade = 'B';

-- Exclude a condition (NOT)
SELECT * FROM students
WHERE NOT grade = 'C';

-- Combine with parentheses for complex logic
SELECT * FROM students
WHERE (grade = 'A' OR grade = 'B')
  AND age < 22;
📌
NULL is special — never compare with =

In SQL, NULL means "unknown". The expression grade = NULL is always false! You must write grade IS NULL or grade IS NOT NULL.

Sorting 05 / 06

Sorting — ORDER BY

Control the order in which rows appear in your results. Without ORDER BY, the database returns rows in an unpredictable order.

sorting.sql ORDER BY 
-- Sort by age, youngest first (ASC = ascending, the default)
SELECT first_name, age
FROM   students
ORDER BY age ASC;

-- Sort by age, oldest first (DESC = descending)
SELECT first_name, age
FROM   students
ORDER BY age DESC;

-- Sort by grade first, then by name (multi-column sort)
SELECT first_name, grade, age
FROM   students
ORDER BY grade ASC, first_name ASC;

-- Sort + filter + limit together
SELECT first_name, age
FROM   students
WHERE  grade = 'A'
ORDER BY age DESC
LIMIT  2;
Output of: ORDER BY grade ASC, first_name ASC
first_namegradeage
PriyaA20
SnehaA19
AnjaliB23
ArjunB22
RahulC21

ASC — Ascending (default)

Numbers: 1 → 9 → 100
Text: A → B → Z
Dates: oldest first

DESC — Descending

Numbers: 100 → 9 → 1
Text: Z → B → A
Dates: newest first

💡
Top-N queries — a very common pattern

Combine ORDER BY with LIMIT to get the top or bottom records. Example: ORDER BY age DESC LIMIT 3 gives you the 3 oldest students. This pattern shows up constantly in analytics.

Joins 06 / 06

Joins — Combining Multiple Tables

Real databases spread data across many tables. Joins let you combine them into one result — like a spreadsheet VLOOKUP but far more powerful.

Let's create a second table — courses — and a linking table — enrollments.

setup_join_tables.sql Setup 
CREATE TABLE courses (
    course_id   INT         PRIMARY KEY,
    course_name VARCHAR(100) NOT NULL,
    instructor  VARCHAR(100)
);

INSERT INTO courses VALUES
    (101, 'SQL Basics',       'Prof. Reddy'),
    (102, 'Python for Data',  'Prof. Kumar'),
    (103, 'Data Warehousing', 'Prof. Iyer');

CREATE TABLE enrollments (
    student_id INT REFERENCES students(student_id),
    course_id  INT REFERENCES courses(course_id),
    enrolled_on DATE,
    PRIMARY KEY (student_id, course_id)
);

INSERT INTO enrollments VALUES
    (1, 101, '2024-01-10'),
    (2, 101, '2024-01-10'),
    (3, 102, '2024-01-11'),
    (1, 103, '2024-01-12');

The 4 join types

INNER JOIN

Only rows that match in both tables. Most common.

LEFT JOIN

All rows from left table; matching rows from right (NULL if no match).

RIGHT JOIN

All rows from right table; matching rows from left (NULL if no match).

FULL OUTER JOIN

All rows from both tables. NULL where no match exists.

CROSS JOIN

Every row in table A paired with every row in table B (cartesian product).

INNER JOIN — the most used join

inner_join.sql INNER JOIN 
-- Which students are enrolled in which courses?
SELECT
    s.first_name,
    s.last_name,
    c.course_name,
    e.enrolled_on
FROM       students    AS s
INNER JOIN enrollments AS e  ON s.student_id = e.student_id
INNER JOIN courses     AS c  ON e.course_id  = c.course_id
ORDER BY   s.first_name;

LEFT JOIN — including unmatched rows

left_join.sql LEFT JOIN 
-- Show ALL students, even those not enrolled in any course
SELECT
    s.first_name,
    c.course_name
FROM      students    AS s
LEFT JOIN enrollments AS e  ON s.student_id = e.student_id
LEFT JOIN courses     AS c  ON e.course_id  = c.course_id;
-- Rahul (id=4) has no enrollment → course_name will show NULL
🧠
When to use which join?

Use INNER JOIN when you only want rows that exist in both tables.
Use LEFT JOIN when you want everything from the left table, even with no match.
Use FULL OUTER JOIN rarely — useful for finding mismatches on both sides.

Table aliases — keeping queries readable

When joining multiple tables, use short aliases (like s for students) so you don't have to type the full table name everywhere.

❌ Without aliases (verbose)

SELECT students.first_name, courses.course_name FROM students INNER JOIN enrollments ON students.student_id = enrollments.student_id

✅ With aliases (clean)

SELECT s.first_name, c.course_name FROM students AS s INNER JOIN enrollments AS e ON s.student_id = e.student_id

Quick Cheat Sheet

Everything you learned today — one reference card.

The Golden Rule of SQL

Read SQL queries in this mental order: FROM (which table?) → WHERE (filter rows) → GROUP BY (group them) → HAVING (filter groups) → SELECT (pick columns) → ORDER BY (sort) → LIMIT (how many). This is the logical processing order, even though SELECT is written first.

DDL

  • CREATE TABLE name (...)
  • ALTER TABLE ADD COLUMN
  • ALTER TABLE DROP COLUMN
  • DROP TABLE name
  • TRUNCATE TABLE name

DML

  • INSERT INTO ... VALUES
  • UPDATE ... SET ... WHERE
  • DELETE FROM ... WHERE

DQL

  • SELECT col FROM table
  • SELECT DISTINCT col
  • SELECT col AS alias
  • GROUP BY col
  • HAVING condition
  • LIMIT n

Filtering

  • WHERE col = value
  • BETWEEN a AND b
  • IN (v1, v2, v3)
  • LIKE 'pattern%'
  • IS NULL / IS NOT NULL
  • AND / OR / NOT

Sorting

  • ORDER BY col ASC
  • ORDER BY col DESC
  • ORDER BY col1, col2

Joins

  • INNER JOIN ... ON
  • LEFT JOIN ... ON
  • RIGHT JOIN ... ON
  • FULL OUTER JOIN ... ON
  • AS alias shorthand
