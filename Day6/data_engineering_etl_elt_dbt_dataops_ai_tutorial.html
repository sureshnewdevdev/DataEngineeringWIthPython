Trainer explanation: In data engineering, the first job is to bring data from many systems into one place. After that, we clean, transform, validate and load the data so business users can use it for reporting, analytics, AI models and decision-making.

1.1 What is Data Ingestion?

Data ingestion means collecting data from different sources and bringing it into a target system for processing. The source can be an application database, CSV file, API, log file, IoT device, Kafka topic, cloud storage or SaaS application.

Source TypeExampleTypical Ingestion Method
DatabaseOracle, SQL Server, MySQL, PostgreSQLJDBC/ODBC, CDC, export/import, replication
FilesCSV, JSON, Excel, ParquetFile watcher, batch load, cloud object storage
APIPayment API, CRM API, HR APIREST API calls, pagination, incremental pulls
StreamingClickstream, IoT, transactionsKafka, Event Hub, Kinesis, Pub/Sub

1.4 Real-Time Example: Retail Sales Ingestion

A retail organization has 100 branches. Every branch uploads a daily sales file to cloud storage. The data engineering team must ingest those files, validate the format, clean bad records and load clean data into reporting tables.

Python Coding Example: Read and Validate Sales Data

import pandas as pd
from pathlib import Path

input_file = Path("daily_sales.csv")
df = pd.read_csv(input_file)
required_columns = ["store_id", "order_id", "order_date", "amount"]
missing_columns = [col for col in required_columns if col not in df.columns]
if missing_columns:
    raise ValueError(f"Missing columns: {missing_columns}")

df["order_date"] = pd.to_datetime(df["order_date"], errors="coerce")
df["amount"] = pd.to_numeric(df["amount"], errors="coerce")
valid_df = df.dropna(subset=["store_id", "order_id", "order_date", "amount"])
invalid_df = df[df.isna().any(axis=1)]
valid_df.to_csv("clean_sales.csv", index=False)
invalid_df.to_csv("rejected_sales.csv", index=False)
print("Valid records:", len(valid_df))
print("Rejected records:", len(invalid_df))

SQL Loading Example

CREATE TABLE raw_sales (

CREATE TABLE sales_summary AS
SELECT store_id, order_date, SUM(amount) AS total_sales

Trainer explanation: Python is useful for orchestration, file handling, API calls and pre-processing. SQL is powerful for joins, aggregations, filters and warehouse-side transformations. A good data engineer should know when to use Python and when to use SQL.

2.1 Basic ETL Pipeline Steps

  1. Extract: Read data from CSV/API/database.
  2. Validate: Check schema, nulls, duplicates and data types.
  3. Transform: Clean, standardize, calculate and join.
  4. Load: Insert into target table or file.
  5. Audit: Store record counts, rejected records and execution status.

2.3 Real-Time Example: Customer Data ETL

In a banking or retail project, customer records may arrive from CRM, branch systems and digital applications. The pipeline must standardize names, clean emails, remove duplicates and load data into a customer master table.

Python Code: Extract and Transform

import pandas as pd
from sqlalchemy import create_engine

customers = pd.read_csv("customers.csv")
customers["name"] = customers["name"].str.strip().str.title()
customers["email"] = customers["email"].str.strip().str.lower()
customers = customers.drop_duplicates(subset=["customer_id"], keep="last")
customers["processed_at"] = pd.Timestamp.now()

engine = create_engine("sqlite:///training.db")
customers.to_sql("stg_customers", engine, if_exists="replace", index=False)
print("Customer data loaded successfully")

SQL Code: Transform Staging to Final Table

CREATE TABLE IF NOT EXISTS dim_customer (
    customer_id INTEGER PRIMARY KEY,
    name        VARCHAR(100),
    email       VARCHAR(150),
    processed_at TIMESTAMP
);

INSERT INTO dim_customer (customer_id, name, email, processed_at)
SELECT customer_id, name, email, processed_at
FROM stg_customers
WHERE email IS NOT NULL;

2.4 Incremental Load Example

In real projects, we do not always reload all data. We load only new or changed records.

SELECT *

Trainer explanation: DBT helps data teams transform data using SQL in a structured, modular and version-controlled way. Instead of writing random SQL scripts, we create models that depend on each other and produce a clear data lineage.

3.1 What is DBT?

DBT is used mainly for the transformation part of modern ELT. Data is already loaded into a warehouse or lakehouse, and DBT helps transform raw data into clean, business-ready tables.

3.3 Simple DBT Model Example

-- models/staging/stg_orders.sql
SELECT
    order_id,
    customer_id,
    CAST(order_date AS DATE) AS order_date,
    CAST(amount AS DECIMAL(12,2)) AS amount
FROM {{ source('raw', 'orders') }}
WHERE order_id IS NOT NULL;

3.4 Using ref()

The ref() function is used to reference another DBT model. It also creates dependency and lineage between models.

-- models/marts/fct_sales.sql
SELECT customer_id, order_date, SUM(amount) AS total_sales
FROM {{ ref('stg_orders') }}
GROUP BY customer_id, order_date;

3.5 Sources

Sources represent raw tables loaded by ingestion tools. By declaring sources, we can document raw tables and test source freshness.

# models/sources.yml
version: 2
sources:
  - name: raw
    tables:
      - name: orders
      - name: customers

3.6 Tests

DBT tests help validate assumptions about data.

# models/schema.yml
version: 2
models:
  - name: stg_orders
    columns:
      - name: order_id
        tests:
          - not_null
          - unique
      - name: amount
        tests:
          - not_null

3.7 Macro Example

Macros allow us to reuse SQL logic.

-- macros/clean_email.sql
{% macro clean_email(column_name) %}
    LOWER(TRIM({{ column_name }}))
{% endmacro %}
-- models/staging/stg_customers.sql
SELECT
    customer_id,
    customer_name,
    {{ clean_email('email') }} AS email
FROM {{ source('raw', 'customers') }};

3.8 Real-Time Example: Retail Sales DBT Flow

In a retail analytics project, raw orders and customers are loaded into the warehouse. DBT is used to build cleaned staging tables, intermediate order calculations and final sales fact tables.

Diagram: DBT Layering Pattern

raw.orders
→
stg_orders
→
int_order_amounts
→
fct_sales

Staging Model

-- stg_orders.sql
SELECT order_id, customer_id, CAST(order_date AS DATE) AS order_date, amount
FROM {{ source('raw', 'orders') }}
WHERE order_id IS NOT NULL;

Intermediate Model

-- int_order_amounts.sql
SELECT
    order_id,
    customer_id,
    order_date,
    amount,
    CASE
        WHEN amount >= 10000 THEN 'HIGH'
        WHEN amount >= 5000 THEN 'MEDIUM'
        ELSE 'LOW'
    END AS order_value_band
FROM {{ ref('stg_orders') }};

Mart Model

-- fct_sales.sql
SELECT
    customer_id,
    order_date,
    order_value_band,
    COUNT(*) AS total_orders,
    SUM(amount) AS total_sales
FROM {{ ref('int_order_amounts') }}
GROUP BY customer_id, order_date, order_value_band;

4.3 Config-Driven Transformation Example

Configuration-driven pipelines reduce hardcoding. Business rules can be changed in a config file instead of changing source code.

YAML Config

# config/customer_pipeline.yml
source_file: customers.csv
target_table: dim_customer
columns:
  customer_id: int
  name: string
  email: string
  country: string
transformations:
  trim_columns:
    - name
    - email
  lowercase_columns:
    - email
  mandatory_columns:
    - customer_id
    - email

Python Code

import pandas as pd
import yaml

with open("config/customer_pipeline.yml", "r") as file:
    config = yaml.safe_load(file)

df = pd.read_csv(config["source_file"])
for col in config["transformations"]["trim_columns"]:
    df[col] = df[col].astype(str).str.strip()
for col in config["transformations"]["lowercase_columns"]:
    df[col] = df[col].astype(str).str.lower()
mandatory = config["transformations"]["mandatory_columns"]
valid_df = df.dropna(subset=mandatory)
print("Target table:", config["target_table"])
print("Valid records:", len(valid_df))

5.2 Prompt Template for Transformation Generation

You are a senior data engineer.
Create SQL transformation logic using the following mapping rules.

Source table: raw_orders
Target table: fct_sales

Rules:
1. order_id should not be null.
2. order_date should be converted to DATE.
3. amount should be converted to DECIMAL(12,2).
4. If amount >= 10000, order_value_band = 'HIGH'.
5. If amount between 5000 and 9999, order_value_band = 'MEDIUM'.
6. Otherwise order_value_band = 'LOW'.

Return:
1. SQL query
2. Validation checks
3. Explanation of logic

5.3 AI-Generated SQL Example

SELECT
    order_id,
    customer_id,
    CAST(order_date AS DATE) AS order_date,
    CAST(amount AS DECIMAL(12,2)) AS amount,
    CASE
        WHEN CAST(amount AS DECIMAL(12,2)) >= 10000 THEN 'HIGH'
        WHEN CAST(amount AS DECIMAL(12,2)) >= 5000 THEN 'MEDIUM'
        ELSE 'LOW'
    END AS order_value_band
FROM raw_orders
WHERE order_id IS NOT NULL;

5.4 AI-Generated Python Example

import pandas as pd

df = pd.read_csv("raw_orders.csv")
df = df[df["order_id"].notna()]
df["order_date"] = pd.to_datetime(df["order_date"], errors="coerce")
df["amount"] = pd.to_numeric(df["amount"], errors="coerce")

def value_band(amount):
    if amount >= 10000:
        return "HIGH"
    elif amount >= 5000:
        return "MEDIUM"
    return "LOW"

df["order_value_band"] = df["amount"].apply(value_band)
df.to_csv("fct_sales.csv", index=False)
