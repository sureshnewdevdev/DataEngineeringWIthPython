  • How to explain pipeline logic clearly to technical and non-technical stakeholders.
  • How to define and present data contracts.
  • How to connect data engineering issues with business impact.

  • How Claude can help generate summaries, architecture notes, slide narratives, and code explanations.

Core Skill

3. Storytelling for Data Engineers

Storytelling means explaining technical data findings in a clear, structured, and business-relevant way. Data engineers often communicate with business analysts, product owners, data architects, QA teams, BI developers, and managers.

Context

What system, dataset, or pipeline are we discussing?

Problem

What issue was found in the source data or pipeline?

Impact

How does this affect reporting, decisions, or operations?

Action

What rule, fix, or decision is required?

Simple Storytelling Formula

1. We analyzed [source/system/table].
2. We found [issue/pattern].
3. This affects [business report/process/decision].
4. We recommend [pipeline rule/action/owner decision].
5. Expected result is [better quality/faster pipeline/trusted report].
Trainer explanation: Do not say only “there are 500 nulls.” Say “500 orders do not have CustomerId, so customer-wise sales reports will be incomplete unless we apply a business rule.”
Communication Skill

4. Communicating Source-System Issues

Source-system issues should be explained with evidence, severity, business impact, and recommended action.

Source Issue Communication Template

Issue:
CustomerId is missing in 4.8% of sales records.

Evidence:
- Total records: 100,000
- Null CustomerId records: 4,800
- Affected date range: 2026-05-01 to 2026-05-31

Business Impact:
Customer-level sales reporting and loyalty analysis will be incomplete.

Recommended Action:
Confirm whether null CustomerId represents guest checkout.
If yes, map to "Guest Customer".
If no, send records to exception handling.
Issue TypeHow to ExplainGood Stakeholder Language
Null valuesShow count and affected fields“Some records are incomplete and may affect reporting accuracy.”
DuplicatesShow duplicate key and count“Revenue may be overstated if duplicate orders are not removed.”
Invalid dateShow parsing failure count“Trend reports may show incorrect daily/monthly numbers.”
Schema driftShow missing or extra columns“The source structure changed and may break the pipeline.”
OutliersShow unusual values“Large transaction values need business confirmation before filtering.”
Presentation Thinking

5. Communicating Pipeline Logic

Pipeline logic should be explained as a simple flow, not only as code. Stakeholders should understand what happens to data at each stage.

Pipeline Logic Explanation Example

The pipeline loads raw sales data into Bronze.
It validates required columns such as OrderId, CustomerId, OrderDate, and Amount.
Invalid records are moved to an exception dataset.
Clean records are transformed into Silver.
Aggregated sales KPIs are created in Gold for Power BI dashboards.
Data Contracts

6. Communicating Data Contracts

A data contract defines expectations between data producers and data consumers. It prevents pipeline failures caused by unexpected source changes.

Contract ElementMeaningExample
Dataset NameName of source or target datasetSalesOrders
OwnerTeam responsible for dataSales Application Team
Required ColumnsColumns that must existOrderId, CustomerId, OrderDate, Amount
Data TypesExpected type for each columnAmount = decimal, OrderDate = date
Freshness SLAWhen data should arriveDaily before 1 AM
Quality RulesValidation rulesOrderId unique, Amount > 0
Change ProcessHow schema changes are communicatedNotify 5 business days before change

Sample Data Contract

Dataset: SalesOrders
Owner: Sales Application Team
Frequency: Daily
Expected Arrival: Before 1 AM IST

Required Columns:
- OrderId: string, not null, unique
- CustomerId: string, nullable only for guest checkout
- OrderDate: date, not null
- Amount: decimal, must be greater than 0
- City: string, not null

Schema Change Rule:
Any column addition, removal, or type change must be communicated before deployment.
Best practice: Data contracts reduce production surprises and improve trust between source-system teams and data platform teams.

Stakeholders may not care about technical terms like schema drift or null percentage. They care about business impact.

Visualization helps communicate profiling results quickly. Data engineers do not always need advanced charts; simple bar charts, line charts, and summary tables are enough for most profiling discussions.

Bar Chart: Null Count by Column

import pandas as pd
import matplotlib.pyplot as plt

null_counts = df.isnull().sum()
null_counts = null_counts[null_counts > 0]

null_counts.plot(kind="bar")
plt.title("Null Count by Column")
plt.xlabel("Column")
plt.ylabel("Null Count")
plt.tight_layout()
plt.show()

Pipeline Requirements

9. Communicating Pipeline Requirements

Pipeline requirements should explain source, target, schedule, quality rules, error handling, and ownership.

Pipeline Requirement Template

Pipeline Name: Daily Sales Ingestion

Source:
- System: POS Sales System
- Format: CSV
- Arrival Time: Daily before 1 AM

Target:
- Bronze: Raw sales file
- Silver: Clean sales table
- Gold: Sales summary table

Validation Rules:
- OrderId must be unique
- Amount must be greater than 0
- OrderDate must be valid
- Required columns must exist

Error Handling:
- Invalid records move to exception table
- Schema drift triggers alert
- Pipeline failure sends notification to support team

Business Output:
- Power BI daily sales dashboard
Interview answer: Pipeline requirements should clearly define input, output, frequency, quality rules, error handling, SLA, ownership, and downstream consumers.

10. DataOps Foundations

DataOps applies software engineering practices to data pipelines. It improves collaboration, reliability, versioning, testing, deployment, and monitoring.

Repository Structure

Organize code, configs, tests, docs, and utilities.

Code Versioning

Track changes using Git and meaningful commits.

Naming Conventions

Use consistent names for files, tables, columns, and jobs.

Reusable Utilities

Create common functions for logging, validation, config, and file handling.

Recommended Repository Structure

data-platform-project/
├── README.md
├── requirements.txt
├── config/
│   ├── dev.yaml
│   ├── test.yaml
│   └── prod.yaml
├── src/
│   ├── ingestion/
│   ├── transformation/
│   ├── validation/
│   ├── utilities/
│   └── pipelines/
├── tests/
├── notebooks/
├── docs/
└── deployment/
Trainer explanation: DataOps starts with discipline. Even a small pipeline should have clear structure, naming, versioning, and reusable utilities.
DataOps Basics

11. Code Versioning and Naming Conventions

Versioning helps teams track what changed, who changed it, and why it changed. Naming conventions reduce confusion and make projects easier to maintain.

Good Commit Messages

feat: add sales ingestion pipeline
fix: handle null CustomerId in sales validation
refactor: move common validation logic to utility function
docs: update data contract for SalesOrders
test: add unit tests for schema validation

Naming Convention Examples

ItemGood NamingPoor Naming
Python filesales_ingestion.pyscript1.py
Functionvalidate_required_columns()check()
Tablesilver_sales_orderstable_new
Columnorder_datedt1
Branchfeature/sales-pipelinemybranch
Best practice: Names should explain purpose. Avoid names like final, latest, temp, new, test1, and script2 in production repositories.

12. Debugging Basics for Data Engineers

Debugging means identifying why a pipeline failed or why output data is incorrect.

Debugging AreaQuestionExample Check
InputDid the source arrive?Check file path and record count
SchemaDid columns change?Compare actual vs expected columns
TransformationDid logic remove too many records?Check row count before and after each step
Data QualityDid validation fail?Check validation error report
OutputWas target written correctly?Check output table/file and row count

Debugging with Row Counts

print("Initial rows:", len(df))

df = df.drop_duplicates(subset=["OrderId"])
print("After deduplication:", len(df))

df = df[df["Amount"] > 0]
print("After amount filter:", len(df))

df = df[df["CustomerId"].notnull()]
print("After customer validation:", len(df))
Best practice: Log row counts before and after every major transformation. It helps identify where records were lost.
Reusable Utility Functions

13. Reusable Utility Functions

Reusable utility functions reduce duplicate code across pipelines.

Utility Function Examples

import pandas as pd
import logging

def read_csv_file(file_path):
    logging.info(f"Reading file: {file_path}")
    return pd.read_csv(file_path)

def validate_required_columns(df, required_columns):
    missing = [col for col in required_columns if col not in df.columns]
    if missing:
        raise ValueError(f"Missing required columns: {missing}")
    return True

def log_row_count(df, step_name):
    logging.info(f"{step_name}: {len(df)} rows")

def save_csv_file(df, file_path):
    logging.info(f"Writing file: {file_path}")
    df.to_csv(file_path, index=False)

Using Utilities in a Pipeline

required_columns = ["OrderId", "CustomerId", "OrderDate", "Amount"]

df = read_csv_file("source_sales.csv")
log_row_count(df, "Source loaded")

validate_required_columns(df, required_columns)

df = df.drop_duplicates(subset=["OrderId"])
log_row_count(df, "After deduplication")

save_csv_file(df, "silver_sales.csv")
