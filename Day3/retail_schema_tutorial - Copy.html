Section 01

What Is a Relational Database?

A relational database organises data into tables (relations) of rows and columns. Every table represents one real-world entity or relationship. Data is retrieved and combined using SQL — a declarative language that describes what you want, not how to fetch it.

📐Core Idea — Codd's Relational Model (1970)

Store each fact exactly once. Reference facts by value, not by pointer. Let the query planner decide the physical access path. This separation of logical design from physical storage is why relational databases have survived 50 years of technology change.

The ACID Guarantee — Why Transactions Matter

Every write to a transactional database must satisfy four properties:

PropertyWhat It MeansRetail Example
AtomicityAll-or-nothing. A transaction either fully commits or fully rolls back.An order insert and its matching stock decrement either both succeed or both fail.
ConsistencyEvery committed transaction leaves the DB in a valid state according to all constraints.A foreign key on order.customer_id prevents orders for non-existent customers.
IsolationConcurrent transactions do not see each other's partial work.Two simultaneous checkouts of the last item in stock cannot both succeed.
DurabilityCommitted data survives crashes.A confirmed payment record persists even if the server restarts mid-flight.

Relational vs. Analytical Layers

In a modern data stack the transactional (OLTP) database is the source of truth. Data flows from it into analytical (OLAP) layers — data warehouses, lakes, or marts — where it is restructured for reporting. Good relational design in the source system is the foundation that keeps the analytical layer trustworthy.

💡DataOps Principle #1 — Schema quality compounds

Every modelling mistake in the OLTP schema propagates downstream. A missing NOT NULL today becomes a nullable dimension key in the warehouse tomorrow. Fix early; the cost multiplies with distance from the source.

Section 02

Keys & Integrity Constraints

Primary Keys (PK)

A primary key is the column (or minimal set of columns) that uniquely identifies every row in a table. Rules: never NULL, unique, stable over time.

✅ Natural Key

Derived from the real world (e.g. product_sku, email). Use when the attribute is truly stable and unique. Risk: the real world changes its mind.

✅ Surrogate Key

A system-generated integer or UUID (e.g. customer_id SERIAL). No business meaning. Preferred in most OLTP schemas — immune to external changes.

⚠️Never use mutable data as a PK

Using email as PK for customers breaks every foreign key when a customer changes their email. Always generate a surrogate.

Foreign Keys (FK)

A foreign key in table B references the primary key of table A, enforcing referential integrity — you cannot create an order for a customer who does not exist, and you cannot delete a customer who has orders (unless a cascade rule is defined).

referential_integrity_example.sql SQL 
-- B references A → referential integrity enforced
ALTER TABLE retail.orders
  ADD CONSTRAINT fk_orders_customer
  FOREIGN KEY (customer_id)
  REFERENCES retail.customers (customer_id)
  ON DELETE RESTRICT   -- prevent orphan orders
  ON UPDATE CASCADE;   -- rare with surrogate keys

Other Integrity Constraints

ConstraintPurposeExample
NOT NULLColumn must always have a value.customer_email NOT NULL
UNIQUEAll values in column(s) are distinct (NULLs allowed).UNIQUE (email)
CHECKRow-level business rule validated at write time.CHECK (unit_price > 0)
DEFAULTSupplies a value when the column is omitted on INSERT.created_at DEFAULT now()
EXCLUDEGeneralised uniqueness (PostgreSQL).No overlapping date ranges for a lease.
✅Rule of Thumb — Constraints belong in the DB, not the app

Application-layer validation can be bypassed (direct DB connections, ETL jobs, migrations). Database constraints cannot. Use both, but the DB is the authoritative enforcement point.

Normalisation — 1NF through BCNF

Normalisation is the process of restructuring tables to eliminate data redundancy and update anomalies. Each normal form adds a stricter rule on top of the previous one.

1

First Normal Form (1NF) — Atomic Values & Single-Valued Columns

Rule: Every column contains exactly one value. No repeating groups, no comma-lists, no embedded arrays.

❌ Violates 1NF

bad_orders.sqlSQL
CREATE TABLE bad_orders (
  order_id    INT,
  customer    TEXT,
  products    TEXT,  -- 'SKU1,SKU2,SKU3'
  quantities  TEXT   -- '2,1,3'
);

✅ Satisfies 1NF

good_order_lines.sqlSQL
CREATE TABLE order_lines (
  order_id    INT,
  product_id  INT,
  quantity    INT,
  PRIMARY KEY (order_id, product_id)
);
2

Second Normal Form (2NF) — No Partial Dependencies

Rule: Every non-key column must depend on the entire primary key, not just part of it. (Only matters when the PK is composite.)

⚠️Partial Dependency Example

If order_lines(order_id, product_id) is the PK, but product_name is stored in that table, it depends only on product_id — a partial dependency. product_name belongs in the products table.

3

Third Normal Form (3NF) — No Transitive Dependencies

Rule: No non-key column should depend on another non-key column. Every non-key column depends directly and only on the PK.

❌ Transitive dependency

In an orders table: order_id → store_id → store_city. store_city depends on store_id, not on order_id.

✅ Fix — extract stores

Move store_city (and all store attributes) to a separate stores table. Orders reference store_id via FK only.

B

Boyce-Codd Normal Form (BCNF) — Strict 3NF

Rule: For every non-trivial functional dependency X → Y, X must be a super-key. BCNF catches anomalies that slip past 3NF when there are multiple overlapping candidate keys. For most retail schemas, achieving 3NF is sufficient.

🎯Practical Target for OLTP Systems

Design to 3NF by default. Selectively denormalise (storing a computed value for performance) only with measurement and documentation. Every denormalisation is technical debt that must be justified.

Section 05

Full DDL Walk-through

Below is the complete, production-quality DDL. Read each block carefully — every design decision has been annotated.

v001__initial_retail_schema.sql PostgreSQL DDL 
-- ============================================================
-- Migration: v001__initial_retail_schema.sql
-- Author:    DataOps Training Team
-- Date:      2025-06-10
-- Purpose:   Create normalised retail OLTP schema
-- Ticket:    DATA-101
-- ============================================================

-- Namespace all objects under the 'retail' schema
CREATE SCHEMA IF NOT EXISTS retail;

-- ── 1. CATEGORIES (self-referencing hierarchy) ──────────────
CREATE TABLE retail.categories (
    category_id        SERIAL           PRIMARY KEY,
    category_name      VARCHAR(100)     NOT NULL,
    parent_category_id INT              REFERENCES retail.categories(category_id),
    created_at         TIMESTAMPTZ      NOT NULL DEFAULT now(),
    CONSTRAINT uq_category_name_parent
        UNIQUE (category_name, parent_category_id)
);
COMMENT ON TABLE  retail.categories IS 'Product taxonomy. NULL parent_category_id = root.';

-- ── 2. CUSTOMERS ────────────────────────────────────────────
CREATE TABLE retail.customers (
    customer_id    SERIAL         PRIMARY KEY,
    customer_name  VARCHAR(200)   NOT NULL,
    email          VARCHAR(254)   NOT NULL,
    phone          VARCHAR(30),
    address_line1  VARCHAR(255),
    address_line2  VARCHAR(255),
    city           VARCHAR(100),
    state_province VARCHAR(100),
    postal_code    VARCHAR(20),
    country_code   CHAR(2)        NOT NULL DEFAULT 'US',
    created_at     TIMESTAMPTZ    NOT NULL DEFAULT now(),
    updated_at     TIMESTAMPTZ    NOT NULL DEFAULT now(),
    is_active      BOOLEAN        NOT NULL DEFAULT TRUE,
    CONSTRAINT uq_customer_email UNIQUE (email)
);
COMMENT ON TABLE retail.customers IS 'Registered customers. Soft-delete via is_active.';
COMMENT ON COLUMN retail.customers.country_code IS 'ISO 3166-1 alpha-2.';

-- ── 3. STORES ───────────────────────────────────────────────
CREATE TABLE retail.stores (
    store_id     SERIAL       PRIMARY KEY,
    store_code   VARCHAR(20)  NOT NULL,
    store_name   VARCHAR(200) NOT NULL,
    city         VARCHAR(100) NOT NULL,
    state_province VARCHAR(100),
    country_code CHAR(2)      NOT NULL DEFAULT 'US',
    opened_date  DATE,
    is_active    BOOLEAN      NOT NULL DEFAULT TRUE,
    CONSTRAINT uq_store_code UNIQUE (store_code)
);

-- ── 4. PRODUCTS ─────────────────────────────────────────────
CREATE TABLE retail.products (
    product_id    SERIAL          PRIMARY KEY,
    category_id   INT             NOT NULL
                    REFERENCES retail.categories(category_id),
    sku           VARCHAR(50)     NOT NULL,
    product_name  VARCHAR(300)    NOT NULL,
    unit_price    NUMERIC(12,2)   NOT NULL,
    currency_code CHAR(3)         NOT NULL DEFAULT 'USD',
    is_active     BOOLEAN         NOT NULL DEFAULT TRUE,
    created_at    TIMESTAMPTZ     NOT NULL DEFAULT now(),
    CONSTRAINT uq_product_sku   UNIQUE (sku),
    CONSTRAINT chk_product_price CHECK (unit_price >= 0)
);

-- ── 5. ORDERS ───────────────────────────────────────────────
CREATE TABLE retail.orders (
    order_id      SERIAL        PRIMARY KEY,
    customer_id   INT           NOT NULL
                    REFERENCES retail.customers(customer_id)
                    ON DELETE RESTRICT,
    store_id      INT           NOT NULL
                    REFERENCES retail.stores(store_id),
    order_date    TIMESTAMPTZ   NOT NULL DEFAULT now(),
    status        VARCHAR(30)   NOT NULL DEFAULT 'PENDING',
    total_amount  NUMERIC(14,2),
    currency_code CHAR(3)       NOT NULL DEFAULT 'USD',
    notes         TEXT,
    created_at    TIMESTAMPTZ   NOT NULL DEFAULT now(),
    updated_at    TIMESTAMPTZ   NOT NULL DEFAULT now(),
    CONSTRAINT chk_order_status
        CHECK (status IN ('PENDING','CONFIRMED','SHIPPED','DELIVERED','CANCELLED'))
);

-- ── 6. ORDER LINES ──────────────────────────────────────────
CREATE TABLE retail.order_lines (
    order_id    INT            NOT NULL
                  REFERENCES retail.orders(order_id)
                  ON DELETE CASCADE,
    product_id  INT            NOT NULL
                  REFERENCES retail.products(product_id),
    quantity    INT            NOT NULL,
    unit_price  NUMERIC(12,2) NOT NULL,
    discount_pct NUMERIC(5,2) NOT NULL DEFAULT 0,
    line_total  NUMERIC(14,2) GENERATED ALWAYS AS
                  (quantity * unit_price * (1 - discount_pct / 100))
                  STORED,
    PRIMARY KEY (order_id, product_id),
    CONSTRAINT chk_ol_quantity     CHECK (quantity     >  0),
    CONSTRAINT chk_ol_unit_price   CHECK (unit_price   >= 0),
    CONSTRAINT chk_ol_discount
        CHECK (discount_pct >= 0 AND discount_pct <= 100)
);

-- ── Indexes (common query patterns) ─────────────────────────
CREATE INDEX idx_orders_customer_id   ON retail.orders(customer_id);
CREATE INDEX idx_orders_store_id      ON retail.orders(store_id);
CREATE INDEX idx_orders_order_date    ON retail.orders(order_date);
CREATE INDEX idx_order_lines_product  ON retail.order_lines(product_id);

DataOps Thread — Versioning & Documenting DDL

Schema changes are code. Every change must be tracked, peer-reviewed, testable, and reversible. The practice of treating DDL with the same discipline as application code is a core DataOps principle.

Versioned Folder Structure

db/ ├── migrations/ │ ├── v001__initial_retail_schema.sql # Create tables, PKs, FKs │ ├── v002__add_product_weight.sql # ALTER TABLE products ADD COLUMN │ ├── v003__order_status_enum.sql # Refactor CHECK to ENUM type │ └── v004__customer_loyalty_tier.sql # New feature column ├── seeds/ │ ├── categories_seed.sql │ └── stores_seed.sql ├── views/ │ └── vw_order_summary.sql ├── docs/ │ ├── schema_assumptions.md # ← DataOps document │ ├── data_dictionary.md │ └── er_diagram.png └── README.md # How to run migrations

PartExampleRule
Version prefixv001__Zero-padded 3-digit number. Double underscore separator (Flyway convention).
Verbadd_, create_, alter_, drop_Lead with what the migration does.
Objectproduct_weightSnake_case, lowercase, no abbreviations.
Extension.sqlAlways .sql; tool-specific formats in sub-folders.

Schema Assumptions Document

docs/schema_assumptions.md Markdown 
# Schema Assumptions — retail.*

## Business Rules Encoded
- An order MUST have a customer and a store (both FK NOT NULL).
- order_lines.unit_price records the price AT TIME OF PURCHASE,
  not the current product price. Price changes never alter history.
- Soft-delete pattern: is_active=FALSE; rows are never deleted.
- country_code follows ISO 3166-1 alpha-2 (2-char).
- currency_code follows ISO 4217 (3-char). Default USD.

## Known Gaps / Future Work
- [ ] No inventory / stock table yet (DATA-145)
- [ ] No discount / promotion table (DATA-201)
- [ ] No audit trail / CDC for orders.status changes (DATA-189)
- [ ] Returns / refunds not modelled (DATA-210)

## Denormalisations
- orders.total_amount: redundant with SUM(order_lines.line_total).
  Stored for performance. Must be kept in sync via trigger or app logic.
  Ticket: DATA-198
🤖DataOps Principle — Document the Why, Not Just the What

A schema assumption document captures decisions that cannot be inferred from DDL alone: why unit_price is duplicated in order_lines, why soft-delete is used instead of hard-delete, what business rule drives a CHECK constraint. This document is the first thing a new engineer and an AI reviewer both need.

Section 07

Naming Standards for Engineering Artifacts

Consistent naming reduces cognitive load, enables automated tooling, and makes AI-assisted review far more accurate. Define your standards once, codify them in a linter or review checklist, and enforce them at code review time.

Tables & Columns

  • Lowercase snake_case everywhere
  • Tables: plural nouns (customers, not Customer)
  • PKs: {table_singular}_id (e.g. customer_id)
  • FKs: same name as the PK they reference
  • Booleans: is_ or has_ prefix
  • Timestamps: _at suffix (TIMESTAMPTZ)
  • Dates: _date suffix (DATE)
  • Money: NUMERIC(14,2), always pair with _currency_code

Constraints & Indexes

  • PKs: auto-named by DB or pk_{table}
  • Unique: uq_{table}_{column(s)}
  • FK: fk_{child_table}_{parent_table}
  • Check: chk_{table}_{column}
  • Indexes: idx_{table}_{column(s)}
  • Views: vw_{name}
  • Materialised views: mv_{name}
  • Procedures: sp_{verb}_{noun}
💡Why Named Constraints Matter for Operations

When a constraint violation appears in a production log at 3 am, fk_orders_customer tells you exactly what failed. constraint_1 does not. Named constraints also make ALTER / DROP operations scriptable without guessing auto-generated names.

Section 08

Using AI (Claude) to Review Schema Design

AI models are excellent at catching naming inconsistencies, missing constraints, unnormalised structures, and gaps in documentation. The key is providing enough context for a high-quality review.

What to Provide to Claude

✅ Give Claude

  • Full DDL (all tables in scope)
  • Schema assumptions document
  • Business description (what this system does)
  • Named review goals (normalisation, constraints, naming, gaps)
  • Any known issues or open tickets

❌ Don't Expect Claude To

  • Know your business domain without context
  • Review security / row-level access controls deeply
  • Predict future query patterns without sample queries
  • Replace a senior DBA for high-stakes production systems
  • Know undocumented assumptions in your head

AI Review Prompts — Copy & Use

You are a senior database engineer. Review the following PostgreSQL DDL for a retail order management system. Check for: (1) normalisation violations (1NF through 3NF), (2) missing NOT NULL constraints, (3) missing CHECK constraints, (4) referential integrity gaps, (5) naming convention inconsistencies, (6) indexes that may be needed for common query patterns (orders by customer, orders by date). List each finding with its severity (High / Medium / Low) and a recommended fix. DDL: [paste DDL here]

Here is our schema assumptions document and DDL for retail.orders and retail.order_lines. Identify any business rules described in the assumptions doc that are NOT enforced by the DDL (missing constraints, missing columns, or missing tables). For each gap, suggest the minimal DDL change to close it.

Given this retail schema, generate 5 common analytical queries that a business intelligence team would run (e.g. top products by revenue, monthly order trends per store). For each query, identify whether the current schema and index setup would support it efficiently, and suggest any missing indexes.

Sample AI Review Output

🤖 Claude Schema Review — retail.* v001 — 7 findings

🔴 HIGH — orders.total_amount is a denormalisation without a documented sync strategy. The column stores a value that can be derived from SUM(order_lines.line_total). If this is intentional (performance), document it in schema_assumptions.md and add a trigger or application constraint to keep it in sync. Stale totals will corrupt downstream analytics.


🟡 MEDIUM — order_lines allows the same product to appear only once per order. The composite PK (order_id, product_id) prevents a customer adding the same SKU in two separate line items. This may be intentional but should be explicitly documented. Consider adding a surrogate PK (line_id SERIAL) if lines may be independently split, returned, or updated.


🟡 MEDIUM — orders.status uses a VARCHAR CHECK, not an ENUM type. This is valid but harder to extend. Consider CREATE TYPE retail.order_status AS ENUM (...) for stronger type safety and self-documentation. Migration v003 could address this.


🟢 LOW — products.currency_code has no foreign key to a currencies table. A free-text CHAR(3) can accept invalid codes. Either add a CHECK constraint validating against the ISO 4217 list, or create a retail.currencies reference table.


🟢 LOW — No updated_at trigger for retail.orders. The column exists but auto-update requires a trigger (BEFORE UPDATE) or application responsibility. Undocumented. Recommend adding a Flyway migration with a set_updated_at() trigger function.

🤖AI Review Workflow in Your DataOps Pipeline

Embed AI schema review into your PR process: before merging any migration, paste the DDL diff into Claude with the review prompt above. Add the AI findings as comments on the PR. A junior engineer can run this review in 5 minutes; a senior engineer then validates the findings. This pattern catches 80% of common issues before they reach production.

Section 10

Summary & Session Checklist

🏗️

Relational Foundations

Tables represent entities. ACID guarantees data integrity. Constraints belong in the database, not just the application layer.

🔑

Keys & Integrity

Surrogate PKs beat natural keys for stability. Every FK enforces referential integrity. CHECK, UNIQUE, and NOT NULL eliminate invalid states at the source.

📐

Normalisation

Design to 3NF by default. 1NF: atomic values. 2NF: no partial dependencies. 3NF: no transitive dependencies. Denormalise consciously and document it.

🛍️

Retail Schema Pattern

6 tables: categories → products → order_lines → orders → customers + stores. Order line captures price at time of purchase. Composite PK on order_lines.

📁

DataOps Versioning

DDL is code. Use versioned migration files, schema assumption docs, and a data dictionary. Every change is peer-reviewed and reversible.

🤖

AI-Assisted Review

Provide DDL + context + explicit review goals to Claude. Use AI for first-pass consistency review. A senior engineer validates. Embed in PR workflow.

Pre-flight Checklist — Before Merging Any Schema Change

🧱 Design Quality

  • All tables in 3NF (document any deliberate exceptions)
  • Every column has the correct data type
  • All FKs defined with ON DELETE behaviour
  • NOT NULL on every mandatory column
  • CHECK constraints for enumerations and numeric ranges
  • UNIQUE constraints for natural business keys
  • Indexes on all FK columns and date/status columns

📋 DataOps Discipline

  • Migration file follows versioned naming convention
  • File has header: author, date, ticket, description
  • Rollback section included (or documented why not)
  • schema_assumptions.md updated if new rules added
  • data_dictionary.md updated with new columns
  • COMMENT ON TABLE/COLUMN for all new objects
  • AI schema review run; findings addressed or accepted
🎓Key Takeaway

Good schema design is not about following rules for their own sake — it is about making data reliable, explainable, and analytically useful long after the original engineers have moved on. The combination of normalisation discipline, versioned DDL, and AI-assisted review gives your team compounding quality advantages from day one.
