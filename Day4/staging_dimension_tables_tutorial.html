Data usually comes from systems such as billing applications, CRM, ERP, Excel files, APIs, or databases. The data is first loaded into staging tables, cleaned, transformed, and then moved into dimension and fact tables.

4. What is a Staging Table?

A staging table is a temporary or intermediate table where raw data is first loaded from source systems before applying business transformations.

Why do we need staging tables?

  • To store raw source data safely
  • To validate data before loading into final tables
  • To clean duplicate, missing, or invalid records
  • To compare new data with existing warehouse data
  • To debug ETL/ELT issues easily

Example

A sales file comes from a POS system every day. We first load it into a staging table called stg_sales_orders.

CREATE TABLE stg_sales_orders (
    order_id        VARCHAR(50),
    order_date      VARCHAR(50),
    customer_name   VARCHAR(100),
    product_name    VARCHAR(100),
    store_name      VARCHAR(100),
    quantity        VARCHAR(20),
    unit_price      VARCHAR(20),
    load_date       DATETIME
);
In staging tables, columns are often kept as flexible data types like VARCHAR because raw source data may contain errors or inconsistent formats.

5. What is a Dimension Table?

A dimension table stores descriptive business information. It helps users analyze facts from different perspectives.

Dimension tables usually answer questions like:

  • Who?
  • What?
  • Where?
  • When?
  • Which category?

Common Dimension Tables

Dimension Table Description Example Columns
dim_customer Stores customer information customer_key, customer_id, customer_name, city, state
dim_product Stores product details product_key, product_id, product_name, category, brand
dim_store Stores store details store_key, store_id, store_name, region, city
dim_date Stores calendar details date_key, full_date, month, quarter, year

Example: Product Dimension

CREATE TABLE dim_product (
    product_key     INT IDENTITY(1,1) PRIMARY KEY,
    product_id      VARCHAR(50),
    product_name    VARCHAR(100),
    category        VARCHAR(50),
    brand           VARCHAR(50),
    is_active       BIT,
    created_date    DATETIME
);
Easy Memory Point: Dimension tables contain descriptive information used for filtering, grouping, and reporting.

6. What is a Fact Table?

A fact table stores measurable business events or transactions. It usually contains numeric values and foreign keys pointing to dimension tables.

Fact tables usually answer questions like:

  • How many?
  • How much?
  • What total?
  • What average?

Example Measures

  • Sales amount
  • Quantity sold
  • Discount amount
  • Profit amount
  • Tax amount

Example: Sales Fact Table

CREATE TABLE fact_sales (
    sales_key       BIGINT IDENTITY(1,1) PRIMARY KEY,
    order_id        VARCHAR(50),
    date_key        INT,
    customer_key    INT,
    product_key     INT,
    store_key       INT,
    quantity        INT,
    unit_price      DECIMAL(10,2),
    sales_amount    DECIMAL(12,2),
    load_date       DATETIME,

    FOREIGN KEY (date_key) REFERENCES dim_date(date_key),
    FOREIGN KEY (customer_key) REFERENCES dim_customer(customer_key),
    FOREIGN KEY (product_key) REFERENCES dim_product(product_key),
    FOREIGN KEY (store_key) REFERENCES dim_store(store_key)
);
Easy Memory Point: Fact tables contain numbers and transactions. Dimension tables describe those numbers.

7. Dimension Table vs Fact Table

Point Dimension Table Fact Table
Purpose Stores descriptive data Stores measurable transaction data
Data Type Mostly text/date/category fields Mostly numbers and foreign keys
Example Customer, Product, Store, Date Sales, Orders, Payments
Question Answered Who, what, where, when How much, how many
Size Usually smaller Usually larger

8. What is a Star Schema?

A star schema is a data warehouse design where a central fact table is connected to multiple dimension tables.

                dim_customer
                     |
dim_product ---- fact_sales ---- dim_store
                     |
                  dim_date

It is called a star schema because the design looks like a star, with the fact table in the center and dimensions around it.

9. ETL Loading Steps

Step 1: Load raw data into staging

INSERT INTO stg_sales_orders
SELECT * FROM source_sales_file;

Step 2: Clean and validate staging data

SELECT *
FROM stg_sales_orders
WHERE quantity IS NULL
   OR TRY_CAST(quantity AS INT) IS NULL;

Step 3: Load dimension tables

INSERT INTO dim_product (product_id, product_name, category, brand, is_active, created_date)
SELECT DISTINCT
    product_id,
    product_name,
    category,
    brand,
    1,
    GETDATE()
FROM stg_sales_orders s
WHERE NOT EXISTS (
    SELECT 1
    FROM dim_product d
    WHERE d.product_id = s.product_id
);

Step 4: Load fact table

INSERT INTO fact_sales (
    order_id,
    date_key,
    customer_key,
    product_key,
    store_key,
    quantity,
    unit_price,
    sales_amount,
    load_date
)
SELECT
    s.order_id,
    d.date_key,
    c.customer_key,
    p.product_key,
    st.store_key,
    CAST(s.quantity AS INT),
    CAST(s.unit_price AS DECIMAL(10,2)),
    CAST(s.quantity AS INT) * CAST(s.unit_price AS DECIMAL(10,2)),
    GETDATE()
FROM stg_sales_orders s
JOIN dim_date d ON CAST(s.order_date AS DATE) = d.full_date
JOIN dim_customer c ON s.customer_id = c.customer_id
JOIN dim_product p ON s.product_id = p.product_id
JOIN dim_store st ON s.store_id = st.store_id;

10. Slowly Changing Dimension

A Slowly Changing Dimension, also called SCD, is used when dimension data changes over time.

Example

A customer changes city from Chennai to Bengaluru. Should we overwrite the old city or maintain history?

SCD Type Meaning Example
Type 1 Overwrite old value Customer city updated directly
Type 2 Maintain history with new row Old city and new city both stored with dates
Type 3 Store limited history in extra column Current city and previous city columns
SCD Type 2 is commonly used when historical reporting is important.

12. Common Interview Questions and Answers

Q1. What is a staging table?

Answer: A staging table is an intermediate table used to store raw data from source systems before cleaning, validation, and transformation.

Q2. Why do we not directly load data into fact and dimension tables?

Answer: Source data may contain duplicates, missing values, wrong formats, and invalid records. Staging helps validate and clean data before loading final warehouse tables.

Q3. What is a dimension table?

Answer: A dimension table stores descriptive information such as customer, product, store, and date details. It helps users filter and group reports.

Q4. What is a fact table?

Answer: A fact table stores business transactions and measurable values such as sales amount, quantity, discount, and profit.

Q5. What is the difference between fact and dimension?

Answer: Dimension tables describe business entities. Fact tables store measurable events connected to those dimensions.

Q6. What is a surrogate key?

Answer: A surrogate key is a system-generated key used as the primary key in dimension tables. Example: customer_key.

Q7. What is a natural key?

Answer: A natural key comes from the source system. Example: customer_id from the CRM system.

13. Best Practices

  • Use clear prefixes like stg_, dim_, and fact_.
  • Keep staging data for debugging when required.
  • Use surrogate keys in dimension tables.
  • Load dimensions before fact tables.
  • Validate duplicate and null records in staging.
  • Maintain audit columns such as created_date, updated_date, and load_date.
  • Use consistent SQL formatting and comments.
  • Document business rules clearly.

14. Hands-On Practice

Create a mini data warehouse for retail sales.

  1. Create one staging table: stg_sales_orders
  2. Create four dimension tables: dim_customer, dim_product, dim_store, dim_date
  3. Create one fact table: fact_sales
  4. Insert sample records into staging
  5. Load dimension tables from staging
  6. Load fact table by joining staging with dimensions
  7. Write a report query to show total sales by product category

Sample Report Query

SELECT
    p.category,
    SUM(f.sales_amount) AS total_sales
FROM fact_sales f
JOIN dim_product p ON f.product_key = p.product_key
GROUP BY p.category
ORDER BY total_sales DESC;
