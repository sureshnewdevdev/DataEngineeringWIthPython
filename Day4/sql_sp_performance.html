Avoid SELECT * — Fetch Only What You Need

SELECT * is the single most common performance anti-pattern. It forces SQL Server to read every column, blocks covering index usage, and breaks procedures when table schemas change.

Why SELECT * hurts performance

🏋️

Extra I/O

Reading all columns means more data pages from disk — even when you only display 3 of 40 columns.

🔍

Kills covering indexes

A covering index can satisfy a query entirely from the index. SELECT * forces a Key Lookup back to the clustered index, making the index useless.

💥

Fragile to schema changes

Adding a column to a table silently changes your SP output. Column order changes can break applications relying on positional column reading.

❌ Anti-pattern
bad_select.sqlBad
CREATE PROCEDURE usp_GetOrders
    @CustomerID INT
AS
BEGIN
    -- Pulls all columns including large TEXT/BLOB fields
    SELECT *
    FROM   dbo.Orders
    WHERE  CustomerID = @CustomerID;
END
✅ Best practice
good_select.sqlGood
CREATE PROCEDURE usp_GetOrders
    @CustomerID INT
AS
BEGIN
    SET NOCOUNT ON;
    -- Only the 4 columns the app actually uses
    SELECT
        OrderID
        ,OrderDate
        ,TotalAmount
        ,Status
    FROM   dbo.Orders
    WHERE  CustomerID = @CustomerID;
END
💡
Rule of thumb

If a covering index exists on (CustomerID) INCLUDE (OrderID, OrderDate, TotalAmount, Status), the explicit SELECT will be satisfied entirely from the index — zero clustered index lookups. SELECT * will always require a Key Lookup, negating the index benefit.

SARGable WHERE Clauses

A SARGable (Search ARGument able) predicate lets SQL Server use an index seek instead of a full scan. Non-SARGable predicates wrap the indexed column in a function or calculation — invalidating the index entirely.

Non-SARGable patterns that kill indexes

Anti-patternWhy it's non-SARGableSARGable fix
WHERE YEAR(OrderDate) = 2024Function wraps the column; index on OrderDate unusableWHERE OrderDate >= '2024-01-01' AND OrderDate < '2025-01-01'
WHERE CONVERT(VARCHAR, ID) = '100'Type conversion hides the column from the optimizerWHERE ID = 100 (use correct type)
WHERE UPPER(LastName) = 'SMITH'Function on column prevents seekUse case-insensitive collation, or computed column with index
WHERE Amount * 1.1 > 500Arithmetic on column side invalidates indexWHERE Amount > 500 / 1.1 (move calc to value side)
WHERE LastName LIKE '%son'Leading wildcard forces full scanWHERE LastName LIKE 'John%' (trailing wildcard OK)
sargable_fix.sqlT-SQL
-- ❌ Non-SARGable: function on indexed column → Table Scan
WHERE DATEPART(YEAR, OrderDate) = 2024

-- ✅ SARGable: range on the column directly → Index Seek
WHERE OrderDate >= '2024-01-01'
  AND OrderDate  <  '2025-01-01'

-- ❌ Non-SARGable: leading wildcard
WHERE ProductCode LIKE '%XYZ'      -- full scan

-- ✅ If suffix search needed: Full-Text Index or REVERSE() persisted column
WHERE ProductCode LIKE 'XYZ%'      -- index seek (trailing wildcard)
⚠️
Check your execution plans

Run your SP with SET STATISTICS IO ON and examine the execution plan. A Table Scan or Clustered Index Scan on a large table is almost always a SARGability problem. Seek = fast. Scan = slow.

Page 03 / 12 SET NOCOUNT ON
Not studied
Query Design↑ Medium Impact

SET NOCOUNT ON

Every INSERT, UPDATE, DELETE, and SELECT in SQL Server sends a "rows affected" message back to the client. In a stored procedure that runs hundreds of DML statements, this generates thousands of extra network round-trips.

nocount.sqlT-SQL
CREATE PROCEDURE usp_ProcessOrders
AS
BEGIN
    SET NOCOUNT ON;  -- ← First line of EVERY stored procedure

    -- Without NOCOUNT ON, each statement below sends
    -- a "X rows affected" message to the client.
    -- With NOCOUNT ON, none of them do. 

    UPDATE dbo.Orders  SET Status = 'Processing'
    WHERE  Status = 'Pending';

    INSERT INTO dbo.AuditLog (Event, LogDate)
    VALUES ('Orders processed', GETDATE());

    -- Return value still works normally
    RETURN 0;
END
📌
Impact at scale

In a tight loop or cursor (even when you shouldn't be using one) executing 50,000 DML statements, SET NOCOUNT ON eliminates 50,000 network messages. In heavily loaded systems, this measurably reduces CPU and network overhead. Make it a reflex — first line, always.

Parameter Sniffing — Understand & Solve It

SQL Server compiles a stored procedure's execution plan using the first parameter value it sees. If that value is atypical, the cached plan is optimal for the sniffed value but terrible for all others.

What happens during sniffing

First call: @CustomerID = 1 — this customer has 2 orders → plan uses Index Seek. Cached. Second call: @CustomerID = 99 — this customer has 50,000 orders → reuses the same Seek plan, which is now suboptimal (a Scan would be better). The SP runs slowly for customer 99 but fast for customer 1.

SolutionWhen to useTrade-off
OPTION (OPTIMIZE FOR UNKNOWN)Parameter distribution is unknown or highly variableUses average statistics; not optimal for any specific value
OPTION (RECOMPILE)Plan varies significantly per execution; SP not called very frequentlyRecompiles every call — CPU overhead on high-frequency SPs
Local variable trickLegacy SQL Server versions; simple SPsOptimizer can't use statistics; may choose bad plan
Multiple SPs by data profileClearly bimodal data (e.g. VIP vs standard customers)Code duplication; maintenance overhead
Query Store forced planSQL Server 2016+; known good plan existsPlan may become outdated as data grows
parameter_sniff_fix.sqlT-SQL
-- SOLUTION 1: OPTIMIZE FOR UNKNOWN (recommended first approach)
CREATE PROCEDURE usp_GetCustomerOrders
    @CustomerID INT
AS
BEGIN
    SET NOCOUNT ON;
    SELECT OrderID, OrderDate, TotalAmount
    FROM   dbo.Orders
    WHERE  CustomerID = @CustomerID
    OPTION (OPTIMIZE FOR (@CustomerID UNKNOWN));
END;

-- SOLUTION 2: Statement-level RECOMPILE for problematic queries only
SELECT OrderID, OrderDate
FROM   dbo.Orders
WHERE  CustomerID = @CustomerID
OPTION (RECOMPILE);  -- recompiles this SELECT only, not the whole SP

-- SOLUTION 3: Local variable (use with caution — loses statistics)
DECLARE @LocalID INT = @CustomerID;
SELECT * FROM dbo.Orders WHERE CustomerID = @LocalID;
🔑
Diagnose with Query Store (SQL Server 2016+)

Run: SELECT TOP 10 * FROM sys.query_store_plan ORDER BY avg_duration DESC to find SPs with multiple plans and high variance. A stored procedure showing 10× duration difference between executions with the same parameter type is a classic sniffing victim.

WITH RECOMPILE — When to Use It

Recompilation forces SQL Server to generate a fresh execution plan on every call — optimal for that run's parameters, but at the cost of compilation CPU each time.

✅

Use WITH RECOMPILE when

The SP uses dynamic or highly variable filters · Data distribution is extremely skewed · SP is called infrequently (batch jobs, reports) · A parameter sniffing fix is urgently needed

❌

Do NOT use when

The SP is called hundreds of times per second · The query plan is stable regardless of parameters · Compilation time exceeds query execution time · You're using it to mask a missing index problem

recompile_usage.sqlT-SQL
-- SP-level recompile: every call rebuilds the whole plan
CREATE PROCEDURE usp_DynamicReport
    @StartDate  DATE
    ,@EndDate    DATE
    ,@StoreID   INT = NULL
WITH RECOMPILE   -- Use for infrequently called, highly variable SPs
AS
BEGIN
    SET NOCOUNT ON;
    SELECT s.StoreName, SUM(o.TotalAmount) AS Revenue
    FROM       dbo.Orders  o
    INNER JOIN dbo.Stores  s  ON o.StoreID = s.StoreID
    WHERE o.OrderDate BETWEEN @StartDate AND @EndDate
      AND (@StoreID IS NULL OR o.StoreID = @StoreID)
    GROUP BY s.StoreName;
END;
Page 06 / 12 Always Schema-Prefix Object Names
Not studied
Compilation & Plans↑ Medium Impact

Always Schema-Prefix Object Names

Referencing a table as Orders instead of dbo.Orders forces SQL Server to resolve the schema at runtime — potentially preventing plan cache reuse and adding overhead on every execution.

❌ No schema prefix
no_schema.sqlBad
SELECT OrderID, TotalAmount
FROM   Orders          -- Which schema?
WHERE  CustomerID = 42;

EXEC   usp_ProcessOrder @ID = 1;
✅ Always schema-prefixed
with_schema.sqlGood
SELECT OrderID, TotalAmount
FROM   dbo.Orders        -- Unambiguous
WHERE  CustomerID = 42;

EXEC   dbo.usp_ProcessOrder @ID = 1;
📌
Plan cache benefit

When two users call the same SP, SQL Server can reuse the cached plan only if the compiled object names match exactly. Without schema prefixes, plans may be compiled separately per user's default schema — wasting plan cache memory and CPU on repeated compilations.

Keep Transactions Short & Focused

Every row touched inside a transaction holds a lock until the transaction commits or rolls back. Long transactions = long locks = blocked queries = deadlocks. This is the #1 cause of production performance emergencies.

❌ Long transaction with slow operations
long_txn.sqlDangerous
BEGIN TRANSACTION;
    -- Lock held during all 3 steps!
    UPDATE dbo.Orders SET Status = 'Processing'
    WHERE  OrderID = @ID;

    -- ❌ Slow email API call INSIDE transaction!
    EXEC dbo.usp_SendConfirmationEmail @OrderID=@ID;

    -- ❌ Slow SELECT aggregation INSIDE transaction
    UPDATE dbo.CustomerStats
    SET    TotalSpend = (SELECT SUM(Amount)
                          FROM dbo.Orders
                          WHERE CustID = @CustID);
COMMIT;
✅ Prepare outside, commit fast
short_txn.sqlSafe
-- Pre-compute OUTSIDE the transaction
DECLARE @NewTotal DECIMAL(14,2);
SELECT @NewTotal = SUM(Amount)
FROM   dbo.Orders
WHERE  CustID = @CustID;

-- Short, focused transaction
BEGIN TRANSACTION;
    UPDATE dbo.Orders SET Status='Processing'
    WHERE  OrderID = @ID;

    UPDATE dbo.CustomerStats
    SET    TotalSpend = @NewTotal
    WHERE  CustID = @CustID;
COMMIT;

-- Email OUTSIDE transaction (no locks held)
EXEC dbo.usp_SendConfirmationEmail @OrderID=@ID;
🚨
Never do these inside a transaction

External API calls · Email sending · File I/O · Large SELECT aggregations · User input waits · WAITFOR DELAY · Any operation that can fail for reasons outside SQL Server's control

Avoid Cursors — Use Set-Based Logic

SQL Server is a set-based engine. Processing rows one at a time with a cursor is like driving a truck one wheel at a time. For 10,000 rows, a cursor can be 100× slower than an equivalent UPDATE or window function.

❌ Cursor approach (slow)
cursor_bad.sqlAvoid
DECLARE @OrderID INT, @Amount DECIMAL(14,2);

DECLARE order_cursor CURSOR FOR
    SELECT OrderID, TotalAmount
    FROM   dbo.Orders
    WHERE  Status = 'Pending';

OPEN order_cursor;
FETCH NEXT FROM order_cursor
    INTO @OrderID, @Amount;

WHILE @@FETCH_STATUS = 0
BEGIN
    UPDATE dbo.Orders
    SET    TaxAmount = @Amount * 0.18
    WHERE  OrderID = @OrderID;
    FETCH NEXT FROM order_cursor
        INTO @OrderID, @Amount;
END
CLOSE order_cursor;
DEALLOCATE order_cursor;
✅ Set-based (fast)
set_based_good.sqlFast
-- One statement replaces the entire cursor loop.
-- SQL Server processes all rows simultaneously
-- using its parallel execution engine.

UPDATE dbo.Orders
SET    TaxAmount = TotalAmount * 0.18
WHERE  Status = 'Pending';

-- Running totals: use window function, not cursor
SELECT
    OrderID
    ,TotalAmount
    ,SUM(TotalAmount) OVER (
        PARTITION BY CustomerID
        ORDER BY OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING
             AND CURRENT ROW
    ) AS RunningTotal
FROM dbo.Orders;
💡
When cursors are acceptable

Cursors are justified for: generating dynamic SQL that differs per row · calling a stored procedure once per record when no set-based equivalent exists · administrative tasks on system metadata. Even then, declare them FAST_FORWARD and READ_ONLY to minimize overhead.

Temp Tables vs Table Variables vs TVPs

Choosing the wrong temporary storage type is a common source of surprising slowdowns. Each has different statistics, locking, and scope characteristics.

TypeStatisticsIndexesScopeBest for
#TempTableYes — full statisticsCan add explicit indexesSession / nested SP callsLarge result sets, joins, complex filtering
@TableVariableNo — always 1 row estimateOnly PK/UNIQUEBatch scope onlySmall lookups (<1000 rows), transaction rollback sensitivity
TVP (Table-Valued Parameter)No by defaultRead-onlyParameter to SPPassing a set of rows into an SP from the application
temp_tables.sqlT-SQL
-- ✅ Temp table with explicit index for large sets
CREATE TABLE #FilteredOrders (
    OrderID     INT          NOT NULL
    ,TotalAmount DECIMAL(14,2)
    ,PRIMARY KEY (OrderID)           -- statistics + clustered index
);

INSERT INTO #FilteredOrders
SELECT OrderID, TotalAmount
FROM   dbo.Orders
WHERE  OrderDate >= '2024-01-01';

-- SQL Server now has accurate statistics on #FilteredOrders
-- and will build an optimal plan for the next JOIN
SELECT f.OrderID, c.CustomerName
FROM       #FilteredOrders   f
INNER JOIN dbo.Customers      c  ON f.OrderID = c.CustomerID;

DROP TABLE #FilteredOrders;   -- always clean up explicitly
⚠️
The @TableVariable cardinality trap

SQL Server always estimates a table variable has 1 row (pre-SQL2019) regardless of actual size. A table variable with 100,000 rows will generate a Nested Loop join plan because the optimizer thinks it has 1 row. Use a #TempTable whenever the row count might exceed a few thousand.

Write SP Queries With Index Awareness

An SP is only as fast as the indexes that support it. Writing a perfect stored procedure that hits an unindexed 10M-row table will always be slow. Index design and SP design must happen together.

🎯

Covering Index

Includes all columns needed by the query in the index itself — zero Key Lookups. Most impactful optimization for read-heavy SPs.

📊

Missing Index DMV

sys.dm_db_missing_index_details and sys.dm_db_missing_index_groups list indexes SQL Server wishes existed based on actual query workload.

index_diagnostics.sqlDiagnostics
-- ── Find missing indexes SQL Server recommends ──────────────
SELECT TOP 20
    mid.statement                       AS TableName
    ,migs.avg_total_user_cost
        * migs.avg_user_impact
        * (migs.user_seeks + migs.user_scans)
                                          AS ImprovementMeasure
    ,mid.equality_columns
    ,mid.inequality_columns
    ,mid.included_columns
    ,'CREATE INDEX IX_' +
      OBJECT_NAME(mid.object_id) + '_missing ON ' +
      mid.statement + ' (' +
      ISNULL(mid.equality_columns,'') + ')' +
      CASE WHEN mid.included_columns IS NOT NULL
           THEN ' INCLUDE ('+mid.included_columns+')'
           ELSE '' END    AS CreateIndexStatement
FROM       sys.dm_db_missing_index_groups  mig
INNER JOIN sys.dm_db_missing_index_group_stats migs
           ON migs.group_handle = mig.index_group_handle
INNER JOIN sys.dm_db_missing_index_details  mid
           ON mid.index_handle  = mig.index_handle
ORDER BY   ImprovementMeasure DESC;

-- ── Find unused indexes (overhead without benefit) ───────────
SELECT
    OBJECT_NAME(i.object_id)  AS TableName
    ,i.name                    AS IndexName
    ,ius.user_seeks
    ,ius.user_scans
    ,ius.user_updates          -- maintenance cost with 0 reads = pure overhead
FROM       sys.indexes                      i
LEFT  JOIN sys.dm_db_index_usage_stats     ius
           ON ius.object_id  = i.object_id
          AND ius.index_id   = i.index_id
WHERE  i.type_desc != 'HEAP'
  AND (ius.user_seeks = 0 OR ius.user_seeks IS NULL)
ORDER BY   ius.user_updates DESC;

TRY/CATCH — Correct Error Handling

Poor error handling leaves open transactions, fills the transaction log, and corrupts data silently. Structured TRY/CATCH ensures transactions are always left in a known state.

try_catch_template.sqlT-SQL Template
CREATE PROCEDURE dbo.usp_TransferStock
    @ProductID     INT
    ,@FromWarehouse INT
    ,@ToWarehouse   INT
    ,@Quantity      INT
AS
BEGIN
    SET NOCOUNT ON;
    SET XACT_ABORT ON;  -- auto-rollback on any runtime error

    BEGIN TRY
        BEGIN TRANSACTION;

        UPDATE dbo.Stock
        SET    Qty = Qty - @Quantity
        WHERE  ProductID   = @ProductID
          AND  WarehouseID = @FromWarehouse;

        UPDATE dbo.Stock
        SET    Qty = Qty + @Quantity
        WHERE  ProductID   = @ProductID
          AND  WarehouseID = @ToWarehouse;

        COMMIT TRANSACTION;
    END TRY
    BEGIN CATCH
        -- Only rollback if a transaction is active
        IF XACT_STATE() <> 0
            ROLLBACK TRANSACTION;

        -- Re-raise the error with full context
        THROW;
    END CATCH;
END;
🔑
Always use SET XACT_ABORT ON

SET XACT_ABORT ON ensures that any runtime error automatically triggers a rollback — even if the CATCH block doesn't execute (e.g. a connection drop). Without it, an uncaught error can leave a transaction open, blocking everyone who touches those rows.

Monitoring & Profiling SP Performance

You cannot improve what you cannot measure. These DMVs and tools give you precise, real-time visibility into which stored procedures are consuming the most resources.

sp_monitoring.sqlDMV Queries
-- ── TOP 10 most expensive SPs (by average CPU) ──────────────
SELECT TOP 10
    OBJECT_NAME(qs.object_id)   AS ProcedureName
    ,qs.execution_count
    ,qs.total_worker_time / 1000 AS TotalCPU_ms
    ,qs.total_worker_time
        / qs.execution_count
        / 1000                  AS AvgCPU_ms
    ,qs.total_elapsed_time
        / qs.execution_count
        / 1000                  AS AvgDuration_ms
    ,qs.total_logical_reads
        / qs.execution_count   AS AvgLogicalReads
    ,qs.last_execution_time
FROM   sys.dm_exec_procedure_stats  qs
WHERE  qs.database_id = DB_ID()
ORDER BY AvgCPU_ms DESC;

-- ── Identify blocking sessions right now ────────────────────
SELECT
    blocking_session_id        AS BlockingSpid
    ,session_id                 AS BlockedSpid
    ,wait_type
    ,wait_time / 1000          AS WaitSecs
    ,SUBSTRING(
        st.text
        ,(r.statement_start_offset/2)+1
        ,256
    )                           AS CurrentStatement
FROM       sys.dm_exec_requests   r
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle)  st
WHERE  blocking_session_id > 0;

-- ── Enable Query Store for continuous monitoring (2016+) ─────
ALTER DATABASE YourDatabase
SET QUERY_STORE = ON
    (OPERATION_MODE  = READ_WRITE
    ,MAX_STORAGE_SIZE_MB = 1024
    ,QUERY_CAPTURE_MODE = AUTO);
📊

Query Store

SQL Server 2016+ built-in performance history. Shows plan regressions, top consumers, and forced plan management — no third-party tools needed.

⚡

Execution Plans

Always check actual (not estimated) plans. Look for thick arrows (many rows), yellow warnings, Key Lookups, and spills to tempdb.

📋

SET STATISTICS IO

SET STATISTICS IO ON reports logical reads per table. High logical reads = missing index or poor query design. This is your first diagnostic tool.
